## Что это

Мобильное B2B-приложение для **ООО ТПК «Полимер-Пласт»** (продажа ПВХ-пластиката).
Тема магистерской ВКР: «Разработка мобильного приложения для бизнеса с интеграцией
интеллектуального чат-бота». Цель — повысить уровень клиентского сервиса: каталог,
формирование КП/счёта, отслеживание заказов, программа лояльности, чат-бот.

**Важно:** тема ВКР — именно МОБИЛЬНОЕ приложение + чат-бот. Не предлагать переезд на
чистый веб/Django — это противоречит защите. Один Expo-кодовая база работает и на телефоне
(клиенты), и в браузере (менеджеры).

## Как запустить и проверить

PostgreSQL установлен как служба Windows (`postgresql-x64-17`), БД `polymerplast`,
пользователь `postgres`/`postgres`, порт 5432. **Тип запуска службы — Manual** (по просьбе
пользователя, чтобы не висела в фоне): при загрузке Windows БД сама НЕ стартует. Перед работой
с бэкендом включай её `POSTGRES-ВКЛ.bat` (в корне; запросит UAC), выключай — `POSTGRES-ВЫКЛ.bat`,
статус — `POSTGRES-СТАТУС.bat`. Если бэкенд падает с `P1001 Can't reach database server` —
значит служба остановлена, запусти `POSTGRES-ВКЛ.bat`.

```bash
# Бэкенд (порт 4000)
cd backend && npm install && npm run dev
#   первичная подготовка БД: npx prisma migrate dev && npm run db:setup
#   сквозной тест: npm run smoke   (ожидается 31/31)

# Приложение (Expo, порт 8081)
cd mobile && npm install && npx expo start   # 'w' — веб, QR — Expo Go
```

Или двойным щелчком: **ЗАПУСТИТЬ-ВСЁ.bat** (см. `ЧИТАЙ-МЕНЯ-запуск.txt`).

**Демо-входы:** `client@demo.ru`/`client123` (клиент, скидка 2%), `big@demo.ru`/`client123`
(5%), `new@demo.ru`/`client123` (0%), `manager@polymer-plast.ru`/`manager123` (менеджер).

**Проверка изменений UI:** есть preview-инструмент (Expo web на 8081). Скриншоты иногда
зависают/рендерят узко после ручного ресайза — проверяй раскладку замерами через
`preview_eval` (getBoundingClientRect), это надёжнее.

## Ключевые решения и соглашения

- **Адаптивность по ширине окна** (`useWindowDimensions`), не по платформе:
  - Веб-навигация (верхняя панель) и 3-колоночные раскладки включаются при `width >= 1024`
    (каталог) / `>= 1000` (карточка товара). Иначе — мобильная вёрстка + нижние вкладки.
  - Веб: верхняя панель (`TopBar`) вместо нижних вкладок; клик по логотипу → главный каталог.
  - **Сквозной «Назад»** (как в браузере): у вкладок `backBehavior="history"`, поэтому
    `goBack()` с корня стека возвращает на предыдущую вкладку (в т.ч. карточка → «Запросить
    счёт» → Корзина → Назад → карточка). Кнопку рисует `navigation/AppHeader.js` (зелёный
    хедер, моб/узкий веб; сам скрывается при `width>=1024`), а на широком вебе — единая кнопка
    в `TopBar` (`canBack`/`onBack` из `RootNavigator`). На стартовом экране (Каталог/Заказы)
    кнопки нет.
- **Роли:** `CLIENT`, `MANAGER`. Менеджер: все заказы, статусы, цены, образцы. Загрузка
  документов разграничена (счёт ТК — менеджер; платёжки — клиент).
- **«Только по данным»** (выбор пользователя): НЕТ Избранного, Сравнения, формы
  Гранула/Порошок — этого нет в данных. Фильтры: метки применения, цена ₽/кг, Шор, хрупкость.
- **Цены:** в БД хранятся **за тонну**; в каталоге и карточке товара показываются **за кг**
  (`pricePerKg = pricePerTon/1000`). КП/счёт и корзина пока считают за тонну.
- **Лояльность:** 200 т → 2%, 500 т → 5% за последние 2 года; скидка на всю компанию.
- **Автоотмена 72ч** — `jobs/autocancel.js` (cron каждые 10 мин).
- **Фото продукции:** 10 файлов в `backend/public/products/`, распределяются по товарам
  детерминированно по `id`; в API поле `imageUrl` = `/static/products/N.jpg`. Артикул =
  `externalCode` (1001..1158).
- **Характеристики и метки** генерируются из названия марки в `prisma/productMeta.js`
  (детерминированно). RAL-цвета парсятся из имени; метки — по сериям/ключевым словам.

## Подводные камни (важно для будущих правок)

1. **Шрифт Inter** (`mobile/src/fonts.js`): патчим `Text.render`, **изменяя ВХОДНЫЕ props**
   (добавляем `fontFamily` к `style` до оригинального рендера). НЕ клонировать ВЫХОДНОЙ
   элемент через `cloneElement` — на react-native-web это даёт `<span>` с массивом стилей и
   крашит всё приложение (пустой экран). 800/900 → Inter Bold (снижение жирности заголовков).
2. **RN-web flexbox:** фиксированные ширины колонок в `flexDirection:'row'` ведут себя
   нестабильно. Для многоколоночных раскладок задавай **явные пиксельные ширины** из
   `useWindowDimensions` (как в `CatalogScreen` и `ProductScreen`).
3. **Импорты в StyleSheet:** если убрать импорт (напр. `radius`), а в `StyleSheet.create`
   осталась ссылка — модуль падает при загрузке → **весь экран пустой, без ошибки в консоли**.
   Проверяй ссылки при удалении импортов.
4. **`useWindowDimensions` в preview** может вернуть устаревшую ширину после ручного ресайза;
   в реальном браузере значение корректное. Скриншот-инструмент тоже капризничает — замеряй
   DOM напрямую.
5. **Кириллица в bash-консоли** Windows отображается «кракозябрами» (cp1251), но файлы в
   UTF-8 — это косметика вывода, не баг.
6. **Кастомный `header` в React Navigation** (опция `header`/`screenOptions.header`) RN
   вызывает **как обычную функцию**, инлайня её хуки в родительский `NativeStackView`. Если
   передать туда компонент с хуками напрямую (`header: AppHeader`) и он условно рендерится —
   ломается порядок хуков → **пустой экран** + ошибка «change in order of Hooks». Всегда
   оборачивай в JSX: `header: (props) => <AppHeader {...props} />` (см. `renderAppHeader` в
   `RootNavigator.js`).
7. **Выпадающие списки (`position:absolute`) уходят под соседний контент.** На RN-web `zIndex`
   действует только между соседями в общем контексте наложения, причём «вглубь» он не
   пробивается. Поэтому: (а) для автоподстановки (`AutocompleteField`) подними **родителя-View
   карточки** с полем над следующей карточкой — `<View style={{ zIndex: 20 }}><Card>…</Card></View>`
   (именно обычный `View`: у `Card` **функциональный `style`**, переданный `zIndex` до DOM не
   доходит — см. `CartScreen`); (б) выпадашку в шапке `FlatList` (сортировка каталога) **выноси
   из списка** и рисуй обычным соседом над `FlatList` с `zIndex` — внутри `VirtualizedList`
   карточки перекрывают абсолютную выпадашку, как ни поднимай вложенный `zIndex` (см.
   `CatalogScreen`, веб-ветка).

## Интеграция LLM в чат-бот (СДЕЛАНО — 2026-06-20)

Гибрид «правила + локальная LLM» реализован и проверен сквозно (бэкенд + веб). Решения:

- **Только локальная LLM, без облака.** Причины: приложение должно работать в регионах без
  доступа к Claude/OpenAI; оплата облачного API из РФ недоступна; тема ВКР — «локальный бот,
  без внешних API». **НЕ предлагать облачные LLM** (Claude/OpenAI/Gemini).
- **Движок — Ollama**, модель **`qwen2.5:14b`** (хорошо знает русский, поддерживает tool
  calling). Запасная — `qwen2.5:7b`. Железо пользователя: Ryzen 7 9800X3D / 32 ГБ RAM /
  RTX 4070 Super (12 ГБ VRAM) — 14B целиком помещается в видеопамять.
- **Гибрид + fallback:** правиловый бот (`services/chatbot.js`) остаётся как быстрый путь и
  страховка. LLM подключается ПОВЕРХ (в первую очередь для нетиповых вопросов). При сбое или
  выключенной Ollama бот автоматически отвечает правилами — приложение не «замолкает».
- **Переключатель** в `backend/.env`: `LLM_PROVIDER=rules|ollama` (+ `OLLAMA_URL=http://localhost:11434`).
- **Управление Ollama:** `OLLAMA-ВКЛ.bat` / `OLLAMA-ВЫКЛ.bat` / `OLLAMA-СТАТУС.bat` в корне
  (уже созданы). ВКЛ — запускает сервер, при первом запуске качает и прогревает модель; ВЫКЛ —
  освобождает VRAM. Это управление процессом Ollama, отдельно от `LLM_PROVIDER`.

**Что реализовано:**
- `backend/src/config/env.js`: `llmProvider` (умолч. `rules`), `ollamaUrl`, `ollamaModel` из `.env`
  (`LLM_PROVIDER` / `OLLAMA_URL` / `OLLAMA_MODEL`). Умолчание `rules` — чтобы `smoke` и базовое
  поведение не зависели от запущенной Ollama.
- `backend/src/services/llm.js`: вызов Ollama `/api/chat`. **RAG: пред-выборка** — поиск по
  каталогу (`searchProductsByText`) выполняется по самому сообщению пользователя ДО модели, и
  найденные марки кладутся в контекст системным сообщением. Это важно: модель часто перефразирует
  запрос и теряет точное название марки (напр. «ПЛ-1»), поэтому полагаться только на её вызов
  инструмента нельзя. Инструмент `search_products` оставлен для уточняющих поисков (`coerceQuery`
  чинит случай, когда `query` приходит объектом). System-prompt: только русский (Qwen склонен
  «съезжать» на китайский — жёсткий запрет + `temperature: 0.2` + страховка-переписывание при
  обнаружении CJK). `isOllamaAvailable()` с кэшем 15 с. Форма ответа:
  `{reply, recommendations, escalate, intent, engineUsed}`.
- `chatbot.js → findProductsByName`: ищет и короткие марки («ПЛ-1», «О-40»), не только латиницу.
  Тонкость: подстрока «пл»/«о» есть в каждом «Пластикат…», поэтому короткую марку ищем только
  ВМЕСТЕ с цифрой и пробуем варианты разделителя (`gradeVariants`: в БД встречается и дефис
  «ПЛ-1», и пробел «Iplex 55»). `searchProductsByText` сперва ищет по названию, затем подбирает
  по характеристикам.
- `routes/chat.routes.js`: `POST /chat` принимает `engine` (`rules|ollama`, переопределяет
  умолчание из `.env`); режим `ollama` в `try/catch` с откатом на `botReply`. В ответе `engineUsed`
  и `fellBack`; `engineUsed` пишется в `meta` сообщения. Новый `GET /chat/engine-status`
  (`{defaultProvider, model, ollamaAvailable}`).
- **Переключатель движка в интерфейсе чата** (`screens/ChatScreen.js`), виден всем (клиент и
  менеджер): сегмент «ИИ · Правила» в шапке; по умолчанию ИИ, если Ollama поднята (иначе Правила).
  Под ответом бота — бейдж активного движка («Ответ ИИ · qwen2.5:14b» / «Базовый движок»). От
  авто-режима отказались осознанно (непрозрачность смены поведения) — выбор ручной, но автооткат
  вниз при сбое/выключенной LLM остаётся.
- **Модель на диске D:** `OLLAMA_MODELS=D:\ollama-models` (переменная пользователя через `setx`),
  т.к. на C: было мало места (~54 ГБ). Сам движок Ollama стоит на C:. Перезапуск сервера после
  установки обязателен — иначе авто-стартованный сервер берёт старый путь (C:).
- Проверка: `npm run smoke` = 31/31 (режим `rules` цел); сквозной тест ИИ/правил в вебе — ок.

**Защита ВКР (план демо):** ПК с RTX 4070 — десктоп, принести нельзя → остаётся дома
включённым как сервер. Доступ для устройств комиссии по QR: фронтенд на Vercel/Netlify
(постоянный URL для QR) + бэкенд+Ollama дома через туннель (напр. Cloudflare Tunnel), адрес
прописать перед защитой. **Страховка обязательна:** запись видео/GIF работы LLM (на случай
сбоя связи) + правиловый fallback. Самый безопасный вариант: QR-демо на правилах, а LLM
показывать самому с экрана.
